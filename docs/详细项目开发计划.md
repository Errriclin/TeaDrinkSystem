# 中小型茶饮店会员与进销存一体化管理系统

## 详细项目开发计划

| 文档信息 | |
|---------|--|
| **项目名称** | 中小型茶饮店会员与进销存一体化管理系统 |
| **文档版本** | V1.0 |
| **编制日期** | 2025-03-11 |
| **计划周期** | 约 6 周（可根据实际调整） |

---

## 1. 计划说明

### 1.1 目的

本计划明确项目各阶段的里程碑、任务分解、交付物及时间安排，用于指导开发、测试与部署，并作为项目管控与汇报的依据。

### 1.2 适用范围

项目组全体成员；可根据学校/甲方要求裁剪阶段或调整周期。

### 1.3 假设与约束

- 开发人员具备 Java、Vue、MySQL 基础。
- 开发环境与测试环境由项目组自行准备（MySQL、Redis、Nginx、Node 等）。
- Canal 需能访问 MySQL 并开启 binlog；若环境受限，可先以“定时同步”等方式模拟，再补全 Canal。

## 3. 阶段一：需求与设计（第 1 周）

### 3.1 目标

确认功能与边界，完成数据库与接口设计，为开发提供明确依据。

### 3.2 任务分解

| 任务编号 | 任务名称 | 工作内容 | 负责人 | 产出物 |
|---------|----------|----------|--------|--------|
| T1.1 | 需求细化与确认 | 对照《详细需求说明书》逐条确认，补充字段与异常流程 | 全员 | 需求确认记录/补充说明 |
| T1.2 | 业务流程梳理 | 绘制采购入库、收银下单、会员充值/消费、库存盘点等流程图 | 开发/分析 | 业务流程图 |
| T1.3 | 数据库设计 | 建表（含主外键、索引）、设计触发器、设计视图（如按日+商品销售统计） | 后端 | 《数据库设计说明书》+ 建表 SQL 初稿 |
| T1.4 | API 设计 | 按模块列出 REST 接口（URL、方法、入参、出参） | 后端/前端 | 《API 设计文档》或 Swagger 初版 |
| T1.5 | 前端页面与路由规划 | 确定页面清单、路由结构、菜单与权限对应关系 | 前端 | 页面/路由清单 |
| T1.6 | 评审与定稿 | 组织评审，修订后定稿 | 全员 | 评审记录、定稿文档 |

### 3.3 交付物清单

- 《详细需求说明书》（已提供，可按评审意见修订）
- 《数据库设计说明书》及建表/触发器/视图 SQL 脚本
- 《API 设计文档》或在线 Swagger
- 业务流程图、页面/路由清单

### 3.4 验收标准

- 需求无重大遗漏，数据库表可支持 F1～F8 功能。
- 触发器、视图设计满足课程/项目要求。
- API 列表覆盖主要功能，前后端对接口约定无歧义。

## 4. 阶段二：后端基础与核心业务（第 2～3 周）

### 4.1 目标

搭建 Spring Boot 项目，完成公共组件与核心业务，实现事务、触发器、视图的完整闭环。

### 4.2 第 2 周：项目骨架与基础模块

| 任务编号 | 任务名称 | 工作内容 | 产出物 |
|---------|----------|----------|--------|
| T2.1 | 工程初始化 | Maven 多模块或单模块、pom 依赖（Spring Boot、MyBatis-Plus、MySQL、Redis、Lombok、Validation 等） | 可启动的空工程 |
| T2.2 | 配置文件与基础配置 | application.yml：数据源、Redis、端口、MyBatis-Plus 分页/逻辑删除等 | 配置文件 |
| T2.3 | 公共组件 | 统一返回体、全局异常处理、分页参数与结果封装、常量类 | 公共包代码 |
| T2.4 | 数据库脚本执行 | 执行建表 SQL，必要时初始化基础数据（如默认角色、管理员） | 数据库就绪 |
| T2.5 | 商品与配料模块 | 商品分类、商品、配料、商品-配料配方的 CRUD 与查询接口 | 接口可调通 |
| T2.6 | 采购与库存模块（基础） | 供应商、采购单（创建/审核）、采购明细；库存表查询；入库时更新库存（事务） | 采购与库存基础功能 |
| T2.7 | 库存盘点 | 盘点录入、差异计算、更新库存（事务），记录操作人 | 盘点接口 |

### 4.3 第 3 周：订单事务、触发器与视图

| 任务编号 | 任务名称 | 工作内容 | 产出物 |
|---------|----------|----------|--------|
| T3.1 | 会员与等级模块 | 会员 CRUD、等级配置、储值充值、储值/积分流水记录 | 会员相关接口 |
| T3.2 | 优惠活动模块 | 活动 CRUD、启用/停用、按时间与状态筛选 | 活动接口 |
| T3.3 | 订单与收银业务 | 创建订单（含明细）、按配方扣减库存、更新会员积分/余额，同一事务；校验库存与余额 | 下单接口（含事务） |
| T3.4 | 触发器部署 | 在 MySQL 中编写并部署库存预警触发器（AFTER UPDATE），写入预警表 | 触发器脚本 + 验证 |
| T3.5 | 视图与报表接口 | 创建销售统计视图（如按日+商品）；提供报表查询接口（销售汇总、库存汇总） | 视图 SQL + 报表 API |
| T3.6 | 员工与权限 | 员工、角色、登录（发 Token）、鉴权拦截；接口按角色控制 | 登录与权限 |

### 4.4 阶段二交付物

- 可运行的后端服务，包含 F1～F8 所需接口。
- 订单事务、触发器、视图均实现并可在库内验证。
- 接口文档（Swagger 或 Markdown）更新至与实现一致。

### 4.5 验收标准

- 通过 Postman/curl 可完成：登录、商品/配料/采购/库存/会员/活动/订单的增删改查及报表查询。
- 下单时故意制造库存不足或异常，验证事务回滚。
- 手动更新库存低于安全值后，预警表有记录；报表接口能查到视图数据。

## 5. 阶段三：缓存与 Canal 同步（第 3～4 周）

### 5.1 目标

引入 Redis 缓存高频数据，通过 Canal 监听 MySQL binlog 将变更同步到 Redis。

### 5.2 任务分解

| 任务编号 | 任务名称 | 工作内容 | 产出物 |
|---------|----------|----------|--------|
| T4.1 | 缓存 Key 设计 | 约定 product、stock、member 等 Key 的命名与结构 | 缓存设计说明 |
| T4.2 | Redis 读写封装 | 在业务层或独立服务中封装 Redis 的读/写/失效逻辑（可选：先读缓存再读 DB） | 工具类或 Service |
| T4.3 | Canal 环境 | 安装配置 Canal Server，连接 MySQL，开启 binlog，配置订阅库表 | Canal 可收到 binlog |
| T4.4 | Canal 客户端 | 在项目中集成 Canal 客户端，解析 binlog 事件，识别目标表的 INSERT/UPDATE/DELETE，更新或删除 Redis 对应 Key | 同步逻辑代码 |
| T4.5 | 联调与验证 | 在库中增删改商品/库存/会员，检查 Redis 中数据是否同步 | 验证报告或演示 |

### 5.3 交付物与验收

- 缓存设计说明与相关代码。
- Canal 配置与客户端代码；核心表变更能反映到 Redis。
- 若环境无法部署 Canal，可先用“定时任务从 MySQL 刷到 Redis”的方式替代，并在文档中说明，后续补 Canal。

## 6. 阶段四：前端开发与联调（第 4～5 周）

### 6.1 目标

完成管理后台与收银端页面，与后端接口联调，实现主要业务流程在前端的闭环。

### 6.2 任务分解

| 任务编号 | 任务名称 | 工作内容 | 产出物 |
|---------|----------|----------|--------|
| T5.1 | 前端工程初始化 | Vite + Vue3 + Element Plus + Vue Router + Axios；统一请求封装与错误处理 | 可运行的前端工程 |
| T5.2 | 布局与登录 | 主布局（顶栏+侧栏+内容区）、登录页、路由守卫（未登录跳转登录） | 布局与登录 |
| T5.3 | 商品与配料页面 | 分类、商品、配料、配方的列表与表单页 | 对应页面 |
| T5.4 | 采购与库存页面 | 供应商、采购单创建/列表、库存列表（含低库存标记）、盘点页 | 对应页面 |
| T5.5 | 库存预警页 | 预警列表、已处理/未处理筛选与操作 | 预警页面 |
| T5.6 | 收银页 | 商品选择、购物车、会员搜索、优惠选择、支付方式、提交订单 | 收银页 |
| T5.7 | 会员与活动页 | 会员列表/详情、等级配置、储值充值；活动列表/编辑 | 对应页面 |
| T5.8 | 报表页 | 销售报表（日/月、按商品）、库存报表、盘点历史 | 报表页 |
| T5.9 | 员工与权限页 | 员工列表、角色配置、菜单权限（若做） | 对应页面 |
| T5.10 | 联调与修复 | 前后端联调，修复接口与数据展示问题 | 联调通过 |

### 6.3 交付物与验收

- 前端源码与构建产物（如 dist）。
- 管理后台与收银端可正常完成：登录、基础配置、采购入库、收银下单、会员与活动、报表查看。
- 与《详细需求说明书》中 F1～F8 对应的主要功能均可通过界面操作完成。

## 7. 阶段五：部署与验收（第 5～6 周）

### 7.1 目标

将系统部署到目标环境，通过 Nginx 提供访问，并完成功能与文档验收。

### 7.2 任务分解

| 任务编号 | 任务名称 | 工作内容 | 产出物 |
|---------|----------|----------|--------|
| T6.1 | 后端部署 | 打包 Spring Boot（jar），在服务器上运行；配置数据源、Redis 等 | 后端服务运行中 |
| T6.2 | 前端构建与静态部署 | npm run build，将 dist 放到服务器目录；Nginx 配置 root/index/try_files | 前端可访问 |
| T6.3 | Nginx 反向代理 | location /api 代理到后端服务；配置静态资源与上传目录（若有） | Nginx 配置 |
| T6.4 | Canal 与 Redis | 在部署环境中配置 Canal 与 Redis，确认同步正常 | 部署文档中的 Canal 章节 |
| T6.5 | 部署文档 | 编写《部署说明》：环境要求、安装步骤、配置项、启动顺序 | 《部署说明》 |
| T6.6 | 功能验收 | 按需求说明书与验收标准逐项演示与记录 | 验收记录 |
| T6.7 | 文档整理 | 汇总需求说明书、开发计划、数据库设计、API 文档、部署说明等 | 项目文档包 |

### 7.3 验收标准

- 通过浏览器访问 Nginx 配置的地址，可打开前端并完成登录。
- 收银、采购、库存、会员、报表等核心流程可完整走通。
- 订单事务、触发器、视图、Canal 同步均有据可查或可演示。
- 《部署说明》可使他人按步骤完成部署。
