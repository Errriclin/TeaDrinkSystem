## 1. 引言

### 1.1 编写目的

本说明书旨在明确“中小型茶饮店会员与进销存一体化管理系统”的功能需求、非功能需求及技术约束，作为系统设计、开发与验收的依据，供项目组、甲方及评审方使用。

### 1.2 项目背景

中小型茶饮店普遍存在以下问题：

- **库存管理混乱**：手工记账或简单 Excel，出入库不及时，易出现缺货或积压。
- **销售数据分散**：收银与后台统计脱节，难以掌握每日销量、热销品、毛利等情况。
- **会员运营薄弱**：仅记录手机号，缺少等级、积分、储值及活动策略。
- **权限与操作安全弱**：多人共用账号，操作不可追溯，易出错或被滥用。

本系统面向线下茶饮门店，提供轻量化的一体化管理能力，覆盖采购、库存、销售、会员、收银与报表全流程，解决上述痛点。

### 1.3 术语与缩略语

| 术语/缩略语 | 说明 |
|------------|------|
| 进销存 | 进货、销售、库存管理的统称 |
| 储值 | 会员预存金额，可在消费时抵扣 |
| 积分 | 会员消费或活动获得的虚拟分值，可兑换或升级等级 |
| RBAC | 基于角色的访问控制（Role-Based Access Control） |
| API | 应用程序接口（Application Programming Interface） |

- 项目立项/课程设计任务书
- 相关技术规范（Spring Boot、Vue3、MySQL 等）

## 2. 总体描述

### 2.1 产品定位

本系统为**面向线下茶饮店的轻量化管理系统**，核心定位如下：

- 覆盖采购、库存、销售、会员、收银、报表全流程。
- 解决个体门店管理混乱、库存对账难、会员运营低效等问题。
- 支持单店或少量门店使用，技术栈轻量、部署简单。

### 2.2 用户特征

| 用户角色 | 描述 | 主要操作 |
|---------|------|----------|
| 店长/管理员 | 负责全局配置与报表查看 | 商品/配料/供应商配置、报表、员工与权限管理 |
| 收银员 | 前台点单与收银 | 收银下单、会员查询、优惠应用 |
| 仓管/采购 | 负责采购与库存 | 采购单、入库、库存盘点、预警处理 |

### 2.3 运行环境

- **服务器**：支持 Linux/Windows Server，建议 2 核 4G 及以上。
- **客户端**：现代浏览器（Chrome、Edge、Firefox 等），支持 PC 端管理后台与收银界面。
- **网络**：门店内网或互联网访问后端服务。

### 2.4 设计与实现约束

- 前端：Vue3 + Element Plus。
- 后端：Java、Spring Boot、MyBatis-Plus。
- 数据库：MySQL，需体现事务、触发器、视图等数据库技术点。
- 缓存与同步：Redis + 阿里 Canal（binlog 监听）实现 MySQL 与 Redis 数据同步。
- 部署：后端独立服务 + Nginx 反向代理及静态资源托管。

## 3. 功能需求

### 3.1 功能模块总览

| 模块编号 | 模块名称 | 功能概要 |
|---------|----------|----------|
| F1 | 商品与配料管理 | 商品分类、商品信息、配料、商品-配料配方 |
| F2 | 采购与库存管理 | 供应商、采购单、入库、库存查看、盘点 |
| F3 | 库存预警 | 基于触发器的低库存预警及预警列表 |

| F5 | 会员与储值积分 | 会员信息、等级、储值、积分、流水 |
| F6 | 优惠活动管理 | 活动配置、启用/停用、收银联动 |
| F7 | 销售与库存报表 | 销售统计、库存统计、基于视图的报表 |

| 需求项 | 描述 |

#### 3.2.2 商品信息管理

| F1.4.2 | 支持在商品编辑页维护配方列表：选择配料、录入消耗数量。 |

#### 3.3.2 采购单管理

| 需求项 | 描述 |
|--------|------|

### 3.4 库存预警（F3）

| 需求项 | 描述 |
|--------|------|
| F3.1 | 在数据库中为库存表设置触发器：当某配料库存数量更新后，若当前库存低于安全库存，则自动向预警表插入一条预警记录（配料、当前库存、安全库存、预警时间、状态）。 |

## 4. 非功能需求

### 4.1 性能

- 列表查询响应时间在常规数据量下不超过 3 秒。
- 收银下单操作在事务内完成，单笔订单处理时间不超过 2 秒（不含网络延迟）。

### 4.2 可用性与可靠性

- 核心业务流程（下单、入库、盘点）依赖数据库事务，保证数据一致性。
- 服务异常时应有统一异常处理与友好错误提示，避免暴露内部细节。

### 4.3 安全性

- 用户密码不可明文存储，需加密（如 BCrypt）。
- 采用 Token 鉴权，防止未授权访问。
- 输入需做校验与转义，防范 SQL 注入与 XSS。

### 4.4 可维护性

- 代码结构清晰，分层明确（Controller / Service / Mapper）。
- 关键业务逻辑有注释；数据库表、视图、触发器有简要说明文档。

### 4.5 兼容性

- 前端支持主流现代浏览器；后端基于 Java 8+ 与 Spring Boot 2.x/3.x。
- 数据库使用 MySQL 5.7+ 或 8.x。

## 5. 数据需求概要

- **商品与配方**：商品、分类、配料、商品-配料关系。
- **采购与库存**：供应商、采购单及明细、库存、盘点记录。
- **预警**：库存预警表（由触发器写入）。
- **订单**：订单主表、订单明细。
- **会员**：会员、等级、储值流水、积分流水。
- **营销**：优惠活动及规则。
- **系统**：员工、角色、权限（或角色-菜单关联）。

具体表结构在《数据库设计说明书》中定义；本系统需体现**事务、触发器、视图**等数据库技术点。

## 6. 接口与集成

- 前端与后端通过 REST API 交互，数据格式为 JSON。
- 后端需提供登录、各业务模块的增删改查及报表查询接口。
- Canal 与 MySQL、Redis 的集成：通过订阅 MySQL binlog，将指定表的变更同步到 Redis，具体表与 Key 设计在技术方案中明确。
