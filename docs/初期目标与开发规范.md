|---------|--|

| **文档版本** | V1.0 |
| **说明** | 约定阶段目标、技术选型顺序、后端分层与联调方式，供团队统一执行 |

## 一、初期目标（阶段一）

### 1.1 业务目标

- 以现有 **`front-end`** 页面与接口约定为准，完成 **MySQL 数据库** 与 **Spring Boot 后端**，实现：
  - 登录 / 注册（与前端 `Login.html`、`Register.html` 对齐）
  - 工作台、收银点单、订单、会员、库存、采购、报表等核心接口可联调通过
- **暂不引入 Redis**：缓存与定时同步放到后期进阶阶段再做。

### 1.2 技术目标

- 数据库：**仅 MySQL**，完成建表、必要 **视图**、**触发器**（如库存预警）、核心业务 **事务**（如下单扣原料库存）。
- 后端：**Java 8 + Spring Boot + MyBatis-Plus**，**三层架构**清晰可维护。
- 联调：**Nginx 反向代理** 统一入口（静态前端 + `/api` 转发后端）；本机已安装 Nginx，部署配置在进阶联调阶段落地到 `deploy/`。

### 1.3 工具与环境（已确定）

| 用途 | 工具 |
|------|------|
| 后端开发 | **IntelliJ IDEA** |
| 数据库设计与 SQL | **DataGrip**（连接 MySQL，执行 `db/` 下脚本、查看表与视图） |
| Web 服务器 / 反向代理 | **Nginx**（已安装） |
| JDK | **JDK 8**（与项目约定一致） |

## 二、技术演进顺序（避免一期范围过大）

| 阶段 | 内容 |
|------|------|
| **初期** | MySQL + 后端三层 + 与 `front-end` 联调；Nginx 做静态资源与 API 反向代理 |
| **后期进阶** | 引入 **Redis**（热点数据缓存）；**定时任务**从 MySQL 刷新到 Redis（本项目已确定不采用 Canal） |

初期代码与配置中可为 Redis 预留注释或空配置块，但**不要求**一期部署 Redis。

## 三、前后端联调与 Nginx 约定

### 3.1 原则

- 浏览器只访问 **Nginx 端口**（如 `80`），不直接写死后端 `8080` 到前端生产配置（开发期可在 IDEA 中单独调试后端）。
- **静态资源**：由 Nginx 指向 `front-end` 构建目录或静态根目录（按实际放置方式调整）。
- **API**：统一前缀如 **`/api/`**，Nginx `proxy_pass` 到 Spring Boot（如 `http://127.0.0.1:8080`）。

### 3.2 示例（示意，正式配置放在 `deploy/nginx`）

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
```

具体路径以前端实际 `API_BASE_URL` 与后端 `context-path` 对齐后写入 `deploy/` 示例文件。

## 四、后端 Java 开发规范：三层架构（企业级习惯）

### 4.1 分层定义

后端工程建议包根：`com.teadrink`（或项目统一包名），严格分为：

| 层级 | 包名示例 | 职责 |
|------|----------|------|
| **控制层** | `...controller` | 接收 HTTP 请求与参数校验；**不写业务规则**；返回统一响应体 |
| **业务层** | `...service` / `...service.impl` | **业务逻辑、事务边界**（`@Transactional`）；编排多个 Mapper；领域规则在此 |
| **持久层** | `...mapper` | **MyBatis（MyBatis-Plus）**：`Mapper` 接口 + 必要时 `resources/mapper/*.xml`；仅数据访问，**不写业务流程** |

可选配套（仍属规范的一部分）：

- `...entity` 或 `...domain`：与表或聚合对应的实体
- `...dto` / `...vo`：入参、出参、与前端 JSON 对齐的对象
- `...config`：Spring、MyBatis、跨域、拦截器等
- `...common`：统一返回 `R` / `Result`、错误码、全局异常处理

### 4.2 依赖方向（强制）

- **Controller → Service → Mapper**；禁止 Controller 直接注入 Mapper（除极少数只读查询且团队书面允许的情况外，默认禁止）。
- **Service** 不依赖 `HttpServletRequest` 等 Web 对象，便于单测与复用。

### 4.3 事务与数据库

- **多表一致性**（下单、入库、会员扣款等）必须在 **Service 层** 方法上加 `@Transactional`，默认回滚异常：`@Transactional(rollbackFor = Exception.class)`。
- **视图**：统计类查询优先走视图或只读查询，由 Mapper 调用。
- **触发器**：在 DataGrip / 脚本中维护于 `db/triggers.sql`，与业务代码职责分离。

### 4.4 接口与前端契约

- 与 `front-end/js/index.js` 中已有路径、字段保持一致；变更接口时**同步更新**本文档或 `docs/API说明.md`（若后续单独维护）。
- 统一响应格式建议与前端解析一致（如 `{ success, msg, data }`）。

## 五、数据库与脚本规范（MySQL + DataGrip）

- 所有 DDL、视图、触发器、初始化数据放在项目 **`db/`** 目录，按文件拆分，例如：
  - `schema.sql`：建表
  - `views.sql`：视图
  - `triggers.sql`：触发器
  - `init_data.sql`：初始化数据
- 使用 **DataGrip** 连接同一套库，与 IDEA 中运行的后端使用同一数据库实例，避免环境不一致。
- 表结构以 **`db/ER设计与表结构概要.md`（精简版）** 为准；变更 ER 后先改概要再改 `schema.sql`。

---

## 六、代码与协作习惯（建议）

- **命名**：类名 UpperCamelCase，方法/变量 lowerCamelCase；表名与 ER 文档一致。
- **注释**：对复杂业务（扣库存、算优惠）保留简短说明即可，避免无意义注释。
- **提交**：与功能点对应，说明「做了什么」而非只写 fix。
- **测试**：重要 Service 方法配合 `logs/测试记录.md` 记录联调结果（见 `docs/测试日志说明.md`）。

---

## 六点五、项目内代码目录（强制）

- **后端 Java** 仅放在 **`backend/`**，按 `controller` / `service` / `mapper` 等子包分类，不得在仓库根目录散落 `.java` 文件。
- **前端** 仅放在 **`front-end/`**，按 `html/`、`js/`、`css/`、`assets/` 分类；新增页面或脚本须归入对应子目录。
- **SQL** 仅放在 **`db/`**；**Nginx 等部署示例** 仅放在 **`deploy/`**。
- 完整树形说明与约定见 **`docs/项目目录结构说明.md`**（已含 `backend` 下三层包路径占位，便于在 IDEA 中直接展开修改）。
