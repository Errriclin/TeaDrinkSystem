# 测试日志说明

## 一、目的与范围

- **目的**：在测试过程中持续记录测试情况、输入数据、预期与实际结果，便于排查问题、回归测试和代码修改。
- **范围**：功能测试、接口测试、前后端联调测试、关键业务流程验证（如下单扣库存、会员积分、库存预警等）。

---

## 二、日志文件位置与类型

| 文件 | 路径 | 说明 |
|------|------|------|
| **测试记录（主日志）** | `logs/测试记录.md` | 按条目追加，记录每次测试的详细数据与结果，方便对照修改代码。 |
| **测试结果摘要（自动）** | `logs/test-results.log` | 由自动化测试（如 JUnit 运行）追加写入的简要结果，可选。 |
| **运行日志** | `logs/app.log` | 应用运行时的业务/错误日志，便于与测试记录对照排查。 |

测试时以 **`logs/测试记录.md`** 为主：手工测试或自动化测试后，均在此按下面格式追加一条或多条记录。

---

## 三、测试记录格式说明（`logs/测试记录.md`）

每条记录建议包含以下字段，便于后续根据“实际结果”与“问题描述”定位代码：

| 字段 | 说明 | 示例 |
|------|------|------|
| **测试时间** | 执行该用例的时间 | 2025-03-12 14:30 |
| **测试轮次** | 第几轮测试（如 第一轮 / 回归-1） | 第一轮 |
| **模块** | 功能模块 | 订单/收银、库存、会员、商品 |
| **用例编号** | 唯一编号，便于引用 | TC-ORDER-001 |
| **用例名称** | 简短描述 | 正常下单并扣减库存 |
| **前置条件** | 数据与环境 | 已登录收银员；商品 A 库存 100 |
| **测试步骤** | 操作步骤简述 | 1. 选商品 A 数量 2 2. 提交订单 |
| **测试数据（输入）** | 请求参数、表单内容等 | orderItems: [{productId:1, qty:2}] |
| **预期结果** | 应出现的结果 | 订单生成，库存变为 98 |
| **实际结果** | 真实结果（含报错信息、接口返回） | 订单成功，库存未变化 |
| **通过/失败** | 通过 / 失败 | 失败 |
| **问题描述** | 现象、报错栈、接口 status/body | 库存未扣减，接口返回 200 但 t_stock 未更新 |
| **修改建议/关联代码** | 建议修改的类、方法、文件 | OrderService.createOrder 未扣减库存；检查 t_stock 更新逻辑 |
| **备注** | 截图路径、链接等 | 截图：screenshots/order-001.png |

## 四、如何“自动”更新测试记录

### 4.1 手工测试（推荐先做）

每次执行完一个用例后：

1. 打开 `logs/测试记录.md`。
2. 在文档末尾按“单条记录模板”复制一份，填齐各字段并保存。
3. 若失败，在“问题描述”和“修改建议/关联代码”中写清，便于后续改代码。

### 4.2 自动化测试结果自动写入（可选）

- **后端（Java）**：在集成 JUnit 后，可使用：
  - **JUnit 5 `TestWatcher`** 或 **`TestExecutionListener`**：在 `testFailed`/`testSuccessful` 时，将用例名、异常信息、时间等追加写入 `logs/test-results.log` 或追加到 `logs/测试记录.md` 的表格行。
  - **Maven Surefire**：运行 `mvn test` 后，在 `target/surefire-reports/` 查看 XML 报告；可再写一个小脚本解析 XML 并追加到 `logs/测试记录.md`。
- **前端**：若使用 Vitest/Cypress 等，可在 afterEach 或 reporter 中把结果写入同一目录下的日志文件或汇总后追加到 `logs/测试记录.md`。

这样，“自动更新”即：**自动化测试运行一次，就自动在日志中追加一条或一批记录**。具体实现可在后端/前端工程就绪后按上述方式接入。

---

## 五、使用建议

- **按模块分表或分节**：在 `测试记录.md` 中用二级标题按模块分节（如 `## 订单与收银`），再在每节下按时间顺序追加记录，便于按功能查。
- **失败必填**：失败的用例务必填写“实际结果”“问题描述”“修改建议/关联代码”，方便定位到具体类、方法或 SQL。
- **保留测试数据**：将关键请求参数、响应 body、数据库变更前后值等粘贴到“测试数据”或“实际结果”中，便于复现和修代码。
- **定期归档**：每轮测试结束可复制一份 `测试记录.md` 为 `测试记录-第N轮-日期.md` 做归档，主文件继续追加新记录。
